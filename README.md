## Архитектура MVVM(за что отвечает каждый слой - модуль(папка в проекте))

SDK организован в виде типичной многослойной структуры:

- `data` — источники данных и DTO (сетевые модели), мапперы, реализация репозитория(по сути это “сырые” данные, которые приходят нам извне)
- `domain` — бизнес-модели (простые классы данных Kotlin) и интерфейсы репозитория, используемые остальной частью приложения(это прослойка между “сырыми” данными и тем, что мы показываем пользователю)
- `presentation` — код пользовательского интерфейса: Compose view, ViewModel, состояние пользовательского интерфейса(все, что мы показываем пользователю)
- `di` — привязки модулей Dependency Injection (Hilt)(это зависимости для правильного buildа проекта)
- `sdk`— общедоступные точки входа SDK и конфигурация для приложений, которые интегрируют библиотеку(это то, как приложение будет общаться с нашей библиотекой)

Модуль приложения (`app`) — это демонстрационное приложение, которое показывает, как использовать SDK(там уже подключена наша библиотека, точнее ее Ui модуль `FinancialCalendarView()`)

## Подробное назначение каждого файла

### `data/dto` - модели данных

### RecurrenceRuleDto.kt

Модель получаемых данных(Data Transfer Object) для запланированных операций

Содержит такие поля:

- frequency - частота повторения(тип **`RecurrenceFrequency`**- enum class, значения которого ограничены: `DAILY`, `WEEKLY`, `MONTHLY`, `YEARLY`)
- interval - число, c какой частотой повторяем(например, frequency = DAILY, interval = 2 ⇒ раз в 2 дня)
- untilEpochMs - когда заканчивается повторение

<aside>
💡

Сейчас мы ее же и отображаем в тестовом приложении, но у нас в domain есть модель RecurrencyRule, в которую мы должны преобразовать RecurrenceRuleDto, чтобы получить только то, что нам нужно отображать пользователю(есть шанс, что RecurrenceRuleDto не содержит лишних данных, поэтому мы сможем избавиться от прослойки в виде RecurrencyRule - вопрос на будущее)

</aside>

### TransactionDto.kt

**Основная модель транзакции**, получаемая с сервера.

Поля:

- id: String?
- amount: Long - сумма, хранящаяся в виде целого числа с наименьшей денежной единицей (речь о копейках, например, 12300 ⇒ 123 рубля)). В коде пользовательского интерфейса мы делим на 100, чтобы отобразить рубли.
- type: TransactionType — доход или расход
- category: String - категория
- dateEpochMs: Long - дата совершения операции
- note: String? - заметка
- isPlanned: Boolean - является ли запланированной
- reccurenceRule: RecurrenceRuleDto? - если операция являяется запланированной, то сюда приходит объект RecurrenceRuleDto в виде JSON, описывающий правило, по которому транзакция повторяется

<aside>
🚨

Аннотация **@SerializedName(...)** используется для сопоставления ключей JSON с полями модели.

</aside>

### `data/network` - работа с сетью

### CalendarAPI

Интерфейс для работы с внешней базой данных

Методы:

- **`getAllTransactions()`** — GET-запрос для получения списка транзакций. Возвращает `List<TransactionDto>`
- **`createTransaction(tx: TransactionDto)`** — POST-запрос для создания транзакции на сервере

**Transaction** и **RecurrenceRule** - это модели, с которыми библиотеке *предпочтительней* работать(то есть они уже очищена от ненужных данных и преобразована в модель с такими полями, с которыми удобно работать)

### `sdk` - публичное API для работы с нашей библиотекой

### CalendarConfig

Класс данных **конфигурации** для инициализации SDK в основном приложении.

Поля:

- supabaseAnonKey - ключ, необходимый для доступа к бэкенду(это пока ТЕСТОВОЕ РЕШЕНИЕ, дальше возможно просто будем получать транзакции в виде списка элементов)
- baseUrl — опциональный параметр для переопределения базового URL API
- enableAnalytics - включена ли аналитика
- theme - всякие шрифты, цвета и тд(описано в ТЗ, но сейчас об этом не думаем)

### CalendarEventListener

Интерфейс для **“прослушивания” событий**, произошедших в нашей библиотеке, основным приложением.

Например, пользователь добавил транзакцию, а в приложении мы об этом узнали, и что-то сделали на это действие.

### CalendarSdk

Это объект, который инициализирует нашу библиотеку в основном приложении.

- **init(context, config)** — инициализирует SDK с помощью предоставленного CalendarConfig. В настоящее время устанавливает только SUPABASE_ANON_KEY
- **setListener(listener)** - устанавливает слушатель на события от библиотеки
- **openDemo** - пока не важно

## Пример использования

1. В классе App, который наследуется от Application

```kotlin
val config = CalendarConfig(
supabaseAnonKey = "<YOUR_SUPABASE_ANON_KEY>",
baseUrl = null // optional override
)
CalendarSdk.init(applicationContext, config)
```

1. Подключаем UI библиотеки в UI приложения через FinancialCalendarView()

```kotlin
setContent {
// If your host app uses Hilt, ensure Hilt is set up and host activity is annotated.
FinancialCalendarView()
}
```
